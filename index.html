Your Coding Agent is Secretly a SOTA Roboticist

RHO (Robotics Harness Optimization) turns a tool-using LLM coding agent into a roboticist. Through reflective evolutionary search, it writes and rewrites a robot's control code in simulation until the code itself solves the task, reaching state-of-the-art manipulation from a frozen, readable codebase, with no human demonstrations and no vision-language-action (VLA) model in the control loop.

01The bottleneck

The old way

The old way kept a language model in the loop the entire time the robot moved.

Recent robotics systems let a language model write the robot's code on the fly, then call it again and again to fix mistakes mid-task. This is the line of work known as Code-as-Policies. The catch: dozens of model calls per attempt make it slow, costly, and easy to break, which puts real-time control out of reach.

The old way: a model in the loop

  • Up to 66 LLM calls per attempt
  • Re-reads the camera and re-prompts the model on every miss
  • Corrects itself mid-rollout (multi-turn)
  • Too slow and brittle for real-time control
→

The RHO way: code that already works

  • Zero LLM calls in the control loop (on CaP-Bench)
  • Executes the code once, start to finish (single-turn)
  • Ships a frozen, readable codebase you can inspect
  • Same action primitives, higher success rate
02The flip

The idea

Let a coding agent write and rewrite the robot's code until it works.

RHO (Robotics Harness Optimization) hands the job to a tool-using coding agent and lets it practice in a simulator through reflective evolutionary search: it edits, runs, and debugs whole programs against the robot's reward, keeping the variants that help. The end product is one self-contained codebase, ordinary code (routing, geometry, retries, checks) wrapped around learned perception and grasping. Because the final product is just code, you can read it, debug it, and reuse it. (For experts: this relocates the model's expensive search out of deployment and into a one-time training phase, so what ships is a frozen policy.)

1

Run

A candidate repository runs in the simulator, earning a reward plus detailed feedback on what went wrong.

2

Rewrite

The repository and that feedback go to the coding agent, which mutates many files at once.

3

Keep the winners

A child is kept only if it beats its parent on a paired minibatch of trials (the acceptance gate).

4

Build a library

The best variants are retained as a diverse library, a Pareto frontier over task coverage, not a single greedy chain.

RHO improves the whole codebase at once, not one prompt or function at a time. A failed grasp usually needs several coordinated fixes together: how the object's position is estimated, how a motion is tuned, how retries work, how the words of a task map to geometry. Rewriting the whole codebase changes all of them in one move.

30% → 70%

A single coding agent (single-turn), no matter how long it is allowed to think, tops out at just 30.0% on the held-out tasks. The reflective evolutionary loop is what lifts the same model and tools to 70%. The loop is the method, not just the agent.

Watching the code teach itself

RAI O3DE: when a model stays in the loop, RHO tunes the harness instead

Here the deployment is multi-turn: a language-model agent keeps running while the robot works. Rather than write standalone code, RHO optimizes the agent's harness, its system prompt, tools, and control code. On the hardest held-out split, success nearly doubles from 23.5% to 44.3% (p < 0.001), while using 27% fewer tool calls and 20% less wall-clock time.

Why rewrite the whole codebase

On Robosuite, mutating a full multi-file repository wins with less compute, and far outperforms rigid structured representations (behavior trees and finite-state machines).

What's still hard

Aggregate gains come with per-task tradeoffs. RHO is below the multi-turn baseline on object-swap on LIBERO-PRO (12.2%) and on cube_restack and two_arm_lift on Robosuite, and nut_assembly remains an open challenge for both RHO and the baseline.

Scope. All results are in simulation. We expect the shipped policies to transfer to real robots only after their perception primitives are re-grounded and the constants fit in simulation are re-tuned. Sim-to-real is a stated expectation with explicit conditions, not a demonstrated result. "State of the art" is scoped to Code-as-Policies methods on these benchmarks.

What is RHO, in plain terms?

RHO (Robotics Harness Optimization) is a tool-using LLM coding agent that writes a robot's control code, then improves it by reflective evolutionary search, running and rewriting whole programs in a simulator until the robot succeeds. The end product is a frozen, readable codebase that reaches state-of-the-art manipulation with no human demonstrations and no vision-language-action (VLA) model in the control loop. The "harness" is all the code and settings around the robot that RHO is allowed to rewrite.

Does the robot still call an AI model while it runs?

For the CaP-Bench deployments (Robosuite and LIBERO-PRO), no. The shipped code runs once, start to finish, over low-level building blocks, with no camera-based re-prompting, no mid-task correction, and no model calls in the control loop. The one exception is the RAI O3DE setting, which intentionally keeps a model running during the task: there, RHO improves the harness (instructions, tools, and control code) that the deployed model uses.

How is this different from Code-as-Policies?

Recent Code-as-Policies systems (like CaP-Agent0) write and repair robot code while the robot runs, up to 66 model calls per attempt (up to 11 per turn). RHO does all of that code generation ahead of time, during a training-time search, and then ships a fixed codebase. What it improves is different too: RHO rewrites a whole multi-file codebase rather than a single prompt or function, which lets it coordinate changes across perception, motion, retries, and task understanding in one move.

How does it compare to VLAs like OpenVLA and π0.5?

Vision-language-action models (VLAs) are single networks that map camera pixels straight to robot actions. They are limited to what they saw in training and need many human teleoperation demonstrations. On the LIBERO-PRO changes, OpenVLA scores 0.0% and π0.5 averages 12.83%, while RHO reaches 45.0% with no demonstrations at all, by learning through practice. RHO also produces readable, debuggable code rather than an opaque network.

Has this been tested on real robots?

Not yet. All benchmarks here are in simulation. We expect transfer to real hardware after re-grounding the perception primitives and re-tuning the constants fit in simulation, but that is a stated expectation with explicit conditions, not a demonstrated result.
